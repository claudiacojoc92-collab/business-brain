M2.1 — Website Magic Moment
Type: Engineering Build Spec (first build slice of M2) Status: READY TO BUILD — governed by ADR-007 (LOCKED), scoped under M2 Blueprint Date: 1 July 2026 Slice size: Small. One connector. One reflection. No auth.
Success criterion: a real founder enters a real website URL and says "Okay — it actually read my business."
This slice builds exactly one thing: the website magic moment. No other connector, no OAuth, no publishing. It is the wedge.

1. Objective
A founder enters their website URL. Within ~30 seconds Business Brain fetches the site, extracts grounded evidence, and reflects back a first grounded observation — positioning, offers, voice, themes — that the founder never typed; the deeper synthesis (inferred insights) recomputed from that evidence streams in behind it. (~30s to first grounded reflection; synthesis streams behind.)
Every reflected claim traces to a page Business Brain actually read. Nothing is invented. When the site yields little or fails, the product says so honestly rather than fabricating.
This slice proves the product thesis end-to-end (observed evidence → model → reflection) using the only source with zero external gate.

2. Founder-facing flow
1. Input. One screen, one focused field: "Enter your website and I'll read it." A single URL input and a primary action.
2. Reading (live). On submit, the founder sees Business Brain working — surfaced as real progress, not a spinner: "Reading your homepage… found your About page… reading your services…" Each line reflects an actual page being fetched.
3. Reflection. Business Brain presents what it found — grounded, specific, organized as recognition, not a data dump (copy in §9).
4. The turn. It closes by naming the boundary of what it can see and inviting the founder to declare intent: "That's what I can see from the outside. What are you actually trying to build?" (This hands off to the conversation — the conversation itself is out of scope for M2.1; the handoff line is in scope as the closing beat.)
5. Honest fallbacks. If the site is thin, unreachable, or JS-only, the founder sees a truthful state and a next step (retry / try another URL / upload instead) — never a fabricated read.

3. Backend flow


URL input
  → normalize + validate URL
  → fetch entry page (bounded)
  → discover key pages (nav + sitemap + common paths)
  → fetch key pages (bounded set)
  → per page: extract readable text + metadata + classify page type
  → produce observed evidence fragments (provenance-bearing)
  → append to evidence store
  → recompute Business Model from website evidence (ADR-007 function)
     → inferred claims carry derived_from → observed fragment ids
  → render first reflection ("what I found") from the model
  → emit honest connection state throughout (reading / synced / partial / empty / failed)
The Business Model engine consumes only evidence fragments from the store — it never sees the fetcher or the raw HTML (ADR-007 boundary). The fetcher's only output is evidence.

4. Website connector contract
Implements the standard connector contract (M2 Blueprint §4). For this no-auth slice, most methods are thin:
Method	M2.1 responsibility
authorize()	No-op. Website is unauthenticated. (Present so the connector conforms to the contract from day one.)
capabilities()	Returns { read: true, insights: false, publish: false }. Static for website.
sync(cursor)	Fetch entry URL, discover + fetch bounded key-page set, return raw page payloads. cursor unused in this slice (no incremental sync — deferred).
normalize()	Per page: strip boilerplate, extract main readable text, parse metadata, classify page type into a canonical shape.
produceEvidence()	Emit one or more observed fragments per page, each with full provenance. The only method that writes to the store.
status()	Report live state (§8) as the run progresses.
disconnect()	Delete this founder's website evidence + any stored run state. (Trivial here; real once persistence exists.)
Boundary (inviolable): the connector emits evidence and nothing else. It does not compute the model, does not call the engine, does not reach downstream.

5. Evidence schema used in this slice
Uses the ADR-007 minimum schema. Fields exercised in M2.1:
Field	M2.1 usage
id	Content-addressed (hash of source_url + payload) — free dedupe on re-run.
source	website
platform	host/domain of the site
source_url	exact page URL the fragment came from
confidence_kind	observed for extracted page content; inferred for model-derived claims
occurred_at	page/article publish date if present in metadata, else null (honest about not knowing)
captured_at	fetch timestamp
visibility	public (it's a public website)
payload	normalized content (text, page_type, extracted metadata)
derived_from	null for observed page fragments; required non-null for any inferred claim (e.g. "positioning is X" points to the homepage fragment it was read from)
Honesty gate (enforced in this slice): an inferred fragment with empty derived_from is rejected by the store. This is how the reflection cannot fabricate — every inferred claim must point at the observed evidence that justifies it. (ADR-007 structural honesty, made real here.)
Store: append-only. Minimal implementation is fine (a table never updated in place). Re-running the same URL appends a fresh run; content-addressed ids dedupe identical fragments.

6. Fetch / crawl rules
Bounded, polite, honest. This is the founder's own site they pointed us at — not arbitrary scraping.
* URL normalization: trim, add scheme if missing (default https://, fall back to http:// on failure), validate it resolves. Reject obviously invalid input with a clear message.
* robots.txt: respect it. If disallowed, honest state, no fetch.
* Page budget: hard cap (≈ 8–12 pages) for the first read. The magic moment is about the key pages, not the whole site.
* Depth: shallow (entry + one hop via discovered key links). No deep recursion.
* Timeouts: per-request timeout (e.g. a few seconds) and a total-run timeout (must resolve within the 30-second promise). Partial result on timeout beats hanging.
* Size cap: per-page byte cap; skip oversized assets; text/HTML only (no media download).
* Concurrency: small, bounded parallel fetch to stay within the time budget.
* Key-page discovery, in priority order:
    1. sitemap.xml if present (highest-signal structure)
    2. Primary nav links from the entry page
    3. Common paths as fallback: /about, /services, /products, /pricing, /blog, /contact
* JS-rendered sites: this slice fetches server-rendered HTML only. No headless browser (deferred). If the entry HTML is effectively empty (SPA shell), detect it and return an honest partial/empty state with guidance — do not guess at content that wasn't served.
* Failure isolation: one page failing never fails the run; it becomes a gap. The run succeeds on whatever it legitimately read.

7. Extraction rules
Per fetched page:
* Readable text: extract main content, strip nav/footer/boilerplate. Preserve headings (they carry positioning signal).
* Metadata: <title>, meta description, Open Graph tags, and JSON-LD / schema.org blocks (org name, services, offers — high-yield structured signal).
* Page-type classification: label each page (home, about, services, pricing, blog_post, contact, other) from URL + heading + content cues. Type drives which register the fragment feeds.
* Register mapping (contribution areas, descriptive — not a model redesign):
    * home → Identity, Offers, Voice
    * about → Identity, Voice
    * services/pricing → Offers
    * blog → Voice, Cadence, Themes
* Language: detect; don't assume English. Non-English content is read and passed through, not dropped.
* Empty detection: if a page yields negligible extractable text, mark it contributing-nothing rather than emitting a hollow fragment.
Extraction produces the normalized payload; produceEvidence() wraps each into an observed fragment with provenance.

8. Connection states (this slice)
Subset of the M2 state machine, honest throughout:
State	Meaning in M2.1
Connecting	URL accepted, fetch starting.
Reading	Actively fetching/extracting pages; drives the live progress copy.
Synced	Read complete, evidence stored, model recomputed, reflection ready.
Partial	Some pages read, some failed/skipped (e.g. SPA sections, timeouts). Reflection uses what was read; gaps stated.
Empty	Reached the site but extracted no meaningful content (SPA shell, near-blank site). Contributes no evidence. No reflection is fabricated. Offer upload as next step.
Failed	Could not reach/resolve the URL, robots-disallowed, or hard error. Honest message + retry / try-another-URL.
Empty and Failed are correct outcomes, not bugs. The one forbidden outcome: presenting a reflection when nothing was legitimately read.

9. First-magic-moment copy examples
Illustrative. Every line is generated from the evidence; these show tone and grounding, not fixed strings.
Reading (live):
* "Reading your homepage…"
* "Found your About page — reading it."
* "Looking at your services…"
* "You've got a blog — reading your recent posts."
Reflection (grounded, specific, traceable):
"Here's what I can already see: You position yourself as [actual positioning pulled from the hero]. Your core offer looks like [real service/pricing structure from the services page]. Your voice reads as [the genuine register — e.g. direct and technical, warm and personal]. Lately you've been writing about [the real recurring blog themes]. You seem to be speaking to [the audience the copy actually addresses]."
The turn (closing beat, hands to conversation):
"That's what I can see from the outside. Now tell me the part I can't see — what are you actually trying to build?"
Partial:
"I read your homepage and services, but a few pages didn't load cleanly. Here's what I've got so far — you can point me at anything I missed."
Empty:
"I reached your site but couldn't read much — it looks like the content loads in a way I can't see yet. Want to upload a page or doc instead so I can read your business properly?"
Failed:
"I couldn't reach that URL. Want to check it and try again, or give me a different link?"
Copy rule: specificity with humility. State what's clearly supported; hedge what's thin; never assert what wasn't read.

10. Acceptance criteria
The slice is done when:
1. A founder enters a real URL and receives a first grounded reflection within ~30 seconds on a typical server-rendered site; the inferred synthesis streams in behind it.
2. Every reflected claim traces to a stored observed fragment — verifiable via derived_from on the inferred claims. No untraceable claim can render.
3. Homepage, about, and services/pricing are discovered and read when present.
4. Metadata (title, description, OG, JSON-LD) is extracted when present.
5. Re-running the same URL does not duplicate identical evidence (content-addressed ids).
6. Empty/partial/failed states render honestly — a thin or JS-only or dead site never produces a fabricated reflection.
7. One page failing does not fail the whole run.
8. The Business Model engine reads only from the evidence store (no coupling to the fetcher).
9. An inferred claim with empty derived_from is rejected by the store (honesty gate enforced).
10. The human criterion: in real testing, a founder looking at the reflection says some version of "it actually read my business."

11. Build order
1. URL input + normalization + validation — the front door; get it robust.
2. Fetcher with bounded rules — entry fetch, timeouts, size/page caps, robots.
3. Key-page discovery — sitemap → nav → common paths.
4. Extraction — readable text, metadata, page-type classification.
5. Evidence fragment creation + append-only store — schema §5, content-addressed ids, honesty gate.
6. Business Model recompute from website evidence — the ADR-007 function over these fragments; inferred claims carry derived_from.
7. Reflection rendering — "what I found," grounded in the model.
8. Connection states + live reading copy — wire the state machine and progress surface.
9. Honest fallbacks — partial / empty / failed paths and their copy.
10. End-to-end pass on real sites — tune to the ~30s-to-first-grounded-reflection promise (synthesis streams behind) and the human criterion.
Steps 1–5 are the spine (evidence in the store). 6–7 are the payoff (reflection). 8–9 are the honesty layer. 10 is the tuning that earns the wedge.

12. Tests
Unit
* URL normalization: missing scheme, trailing junk, invalid input, http/https fallback.
* Key-page discovery: sitemap present/absent; nav parsing; common-path fallback.
* Extraction: main-content vs boilerplate; metadata parse (title/description/OG/JSON-LD); page-type classification; language detection.
* Fragment creation: schema conformance; content-addressed id stability; captured_at set; occurred_at null when unknown.
* Honesty gate: inferred fragment with empty derived_from is rejected.
Integration
* Full run on a representative server-rendered site → fragments in store → recompute → reflection, all claims traceable.
* Re-run dedupe: identical content produces no duplicate fragments.
* Engine isolation: model builds purely from store, no fetcher coupling.
Edge / honesty (the important ones)
* Thin one-pager → shallow but honest reflection, no padding.
* SPA / JS-only shell → Empty, upload offered, no fabricated reflection.
* Dead / unresolvable URL → Failed, clean retry.
* robots-disallowed → Failed/blocked, honest, no fetch.
* One key page 500s / times out → run succeeds Partial, gap stated.
* Oversized / sprawling site → respects page + time budget, resolves in time.
* Non-English site → read and reflected, not dropped.
* Empty site → contributes zero evidence; model reflects nothing rather than inventing.
Performance
* Typical site delivers the first grounded reflection within ~30s (synthesis streams behind); measure p50/p95 for both beats.

13. Out of scope
Explicitly not in M2.1:
* Any other connector (Instagram, Google, LinkedIn, upload — all later).
* OAuth / credential infrastructure (no auth in this slice).
* Publishing / write-back (ADR-007 forbidden until read-trust earned).
* The conversation itself / declared-intent capture (only the closing handoff line is in scope).
* Headless-browser / JS rendering (SPAs handled by honest Empty, not by rendering).
* Incremental sync, webhooks, orchestrator, plugin SDK, relational evidence graph (ADR-007 deferred).
* Deep crawling, full-site indexing, third-party/competitor site fetching (forbidden).
* Any redesign of the Business Model engine (ADR-007 — consume evidence, don't redesign).
