Implementation Brief — M2.1 Website Magic Moment
For: Claude Code (senior engineer, repo local) From: CTO / Head of Product Status: APPROVED & FROZEN as the next milestone. Governed by ADR-007 (LOCKED). Date: 1 July 2026 Reference docs: ADR-007 (Connect Your World Nucleus, LOCKED), M2 Blueprint, M2.1 Website Magic Moment Build Spec.
You are an excellent senior engineer with the repository already local. This is a ticket, not a tutorial. Read it fully before touching anything. Do not redesign the product. Do not reopen architecture. When in doubt, prefer the smaller, more honest implementation.

1. Objective
Build the first Connect Your World slice: a founder enters their website URL, and Business Brain fetches the site, extracts grounded observed evidence, recomputes the Business Model from that evidence, and renders a first reflection ("what I found") — specific, and traceable to pages actually read — within ~30 seconds. Empty/partial/failed cases are handled honestly and never fabricate a read.
The bar is a human one: a real founder enters a real URL and says "Okay — it actually read my business."
This slice is the wedge. It has no authentication and touches exactly one connector.

2. Scope
In scope (build exactly this)
* Website URL input, normalization, and validation.
* A Website connector implementing the full standard connector contract (M2 Blueprint §4), with unauthenticated methods thin/no-op where appropriate — but implementing the whole contract, not a website-special-case shortcut.
* Bounded fetch of the entry page and a small set of discovered key pages (fetch rules per M2.1 §6).
* Key-page discovery: sitemap → primary nav → common-path fallback.
* Extraction: readable main text, metadata (title, meta description, Open Graph, JSON-LD), and page-type classification.
* Evidence fragment creation as observed, provenance-bearing, per the ADR-007 minimum schema (M2.1 §5).
* Append-only evidence store — minimal but real: never updated in place, content-addressed ids for dedupe, and the honesty gate (reject inferred fragments with empty derived_from).
* Business Model recompute from website evidence using the existing frozen engine, via its evidence-input contract (see §3 and §6). Inferred claims carry derived_from → observed fragment ids.
* First reflection rendering ("what I found") from the recomputed model.
* Connection states: connecting / reading / synced / partial / empty / failed, surfaced honestly, including the live "reading" progress.
* The closing handoff line only (the sentence that invites declared intent). See "not in scope" for the conversation itself.
NOT in scope (do not build — no exceptions)
* Any other connector: Instagram, Facebook, LinkedIn, Google, YouTube, upload, X, TikTok, etc.
* OAuth / credential infrastructure. Website is unauthenticated. Do not build the shared OAuth layer in this slice.
* Publishing / write-back of any kind.
* The conversation / declared-intent capture flow. Only the static closing handoff line is in scope; capturing and storing the founder's reply is a later slice.
* Headless browser / JS rendering. Server-rendered HTML only. SPA shells resolve to honest empty, not to rendering.
* Account system as a prerequisite. Use the minimum founder/session scaffolding required to attach evidence and render a reflection. If the slice can be validated without full accounts, do so.
* Any change to the Business Model engine's internal logic. You may add a thin input adapter if — and only if — the engine cannot already consume evidence-shaped input (see §6). You may not redesign, refactor, or "improve" the engine.
* Incremental sync, webhooks, orchestrator, plugin SDK, relational evidence graph — all ADR-007 deferred.
* Deep crawling, full-site indexing, or fetching any site other than the one the founder entered. No third-party/competitor fetching.
If a task seems to require something in the "not in scope" list, stop and flag it rather than building it.

3. Existing architecture you must respect
* ADR-007 is frozen. The principle is: Business Brain learns from connected reality (observed) and declared intent (declared); it never asks founders to reconstruct what it can fetch. Reconstruction is forbidden. Read ADR-007 before writing anything; it defines the evidence schema, the honesty guarantees, and the forbidden list.
* The Business Model engine is frozen. It already exists (built in M1, covered by an existing Vitest suite). Its job — turning inputs into the Business Model — is not yours to change. Your slice feeds it evidence and renders its output. Treat it as a fixed downstream consumer.
* The evidence boundary is inviolable. The engine consumes only evidence from the store. The Website connector's only output is evidence. The connector must not call the engine, must not compute the model, and must not reach downstream. Fetcher → evidence store → engine → reflection, one direction.
* The honesty model is structural, not advisory. observed fragments must carry a real source. inferred fragments must carry non-empty derived_from. A claim that cannot be traced to evidence must not be able to render. This is the M1 fabrication fix; do not weaken it.
* Architecture/Engine Freeze conventions apply. You are the sole agent touching this repo. Respect existing structure, naming, prompt/architecture freezes, and the existing test discipline (Vitest). Extend the codebase in its own idiom; do not restructure it.
You are not authorized to reopen or reinterpret any of the above. If the existing architecture appears to conflict with this brief, surface the conflict — do not resolve it by redesigning.

5. Implementation plan (engineering approach, no code)
Step 0 — Contract spike (do this first). Confirm how the frozen Business Model engine consumes input today. Two outcomes: (a) it already accepts evidence-shaped input → feed it directly; (b) it accepts the manual-input shape → build a thin, well-tested adapter that maps evidence fragments to the engine's existing input contract, changing nothing inside the engine. Decide and document which path before proceeding. If the seam is larger than a thin adapter, stop and flag — do not redesign the engine.
Step 1 — URL front door. Input, normalization (scheme defaulting, trimming, validation), and resolvability check. Robust handling of malformed input with clear, honest error surfaces.
Step 2 — Fetcher with bounded rules. Entry-page fetch under the M2.1 §6 budget: robots.txt respect, page cap (~8–12), shallow depth, per-request and total-run timeouts (must fit the 30-second promise), size caps, text/HTML only, bounded concurrency, per-page failure isolation.
Step 3 — Key-page discovery. sitemap.xml → primary nav links → common-path fallback (/about, /services, /products, /pricing, /blog, /contact).
Step 4 — Extraction + classification. Main readable text (strip boilerplate, keep headings), metadata (title/description/OG/JSON-LD), page-type classification, language detection, and empty-page detection.
Step 5 — Evidence store + fragment creation. Append-only store (never update in place), content-addressed ids, ADR-007 minimum schema, and the honesty gate enforced at the store (reject inferred with empty derived_from). produceEvidence() emits observed fragments with full provenance; occurred_at from page metadata or null.
Step 6 — Model recompute. Feed the stored website evidence to the frozen engine (directly or via the Step 0 adapter). Ensure any inferred output carries derived_from pointing at the observed fragments it was reasoned from.
Step 7 — Reflection rendering. Render the "what I found" reflection from the model, grounded and specific, every claim traceable. Include the closing handoff line. Respect the existing design system.
Step 8 — States + live reading + honest fallbacks. Wire the connection-state machine, the live "reading" progress surface, and the partial/empty/failed paths with their honest copy. Confirm no path can render a reflection when nothing was legitimately read.
Step 9 — End-to-end tuning. Run against a spread of real sites; tune to the 30-second budget and the human criterion; verify honesty behavior on thin/SPA/dead/non-English sites.
Steps 1–5 are the spine (real evidence in the store). 6–7 are the payoff. 8 is the honesty layer. Build the spine before the payoff so the foundation is verified before the founder-facing surface.

6. Risks — where you're most likely to violate ADR-007, and how not to
1. Redesigning the engine instead of adapting to it. The seam between evidence fragments and the M1 input contract will tempt refactoring. Guardrail: an adapter is allowed; touching engine internals is not. If a thin adapter can't bridge it, stop and flag — do not "improve" the engine.
2. Fabricating a reflection when the site is thin/SPA/empty. The magic moment creates pressure to always say something impressive. Guardrail: the honesty gate is enforced at the store; empty/partial/failed must render honestly; no claim without traceable evidence. A boring-but-true "I couldn't read much — upload instead" is a correct outcome, not a failure.
3. Dropping provenance on inferred claims. Rendering "your positioning is X" without linking it to the page it came from silently reintroduces the M1 fabrication class. Guardrail: inferred fragments require non-empty derived_from; the store rejects otherwise; the reflection renders only traceable claims.
4. Building the connector as a website special-case. Shortcutting the contract (skipping capabilities()/status()/disconnect()) breaks the five-year plug-in design. Guardrail: implement the full contract, thin where unauthenticated, so the next connector slots into the same shape.
5. Scope creep toward upload/OAuth/conversation. Adjacent needs will surface (a thin site "wants" upload; the handoff line "wants" to capture the reply). Guardrail: stop at the boundary. The handoff line is static; capturing the reply is a later slice. Upload is a later slice. Flag, don't build.
6. Crawling beyond the founder's own site. Following external links or deep-crawling violates the forbidden list. Guardrail: fetch only the entered site, bounded; no third-party fetching.
7. Introducing browser storage / new persistence patterns. Guardrail: reuse existing store conventions; append-only; no ad-hoc caches that could serve stale data as fresh reads.
8. Deleting or repurposing the M1/M1.5 harness. It remains as an internal validation harness (ADR-007). Guardrail: leave it intact; you are adding the website path, not removing the harness.

7. Acceptance criteria (M2.1 is complete when all are true)
1. A founder enters a real URL and receives a grounded reflection within ~30 seconds on a typical server-rendered site (measure and report p50/p95).
2. Every reflected claim is traceable to a stored observed fragment via derived_from. No untraceable claim can render.
3. Homepage, about, and services/pricing are discovered and read when present; metadata (title/description/OG/JSON-LD) extracted when present.
4. Re-running the same URL produces no duplicate evidence (content-addressed ids).
5. empty, partial, and failed states render honestly; thin/SPA/dead sites never produce a fabricated reflection.
6. One page failing does not fail the run; the run succeeds on what was legitimately read (partial, gap stated).
7. The Business Model engine reads only from the evidence store; the connector has no downstream coupling. Engine internals unchanged (adapter, if any, is thin and isolated).
8. The store rejects an inferred fragment with empty derived_from (honesty gate proven by test).
9. The full existing Vitest suite still passes; new tests (§ deliverables) pass.
10. Human criterion: in real testing against real founder sites, the reflection earns some version of "it actually read my business."
